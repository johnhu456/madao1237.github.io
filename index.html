实现OC中的可变参数传递的方法 

- (id)performSelectorWithArgs:(SEL)sel, ..
{
INIT_INV(sel, nil);
 [inv invoke];
 return [NSObject getReturnFromInv:inv withSig:sig];
}

INIT_INV(sel,nil)

一个宏函数，使用了C标准的stdarg.h库。
内容如下 

#define INIT_INV(_last_arg_, _return_) \
NSMethodSignature * sig = [self methodSignatureForSelector:sel]; \
if (!sig) { [self doesNotRecognizeSelector:sel]; return _return_; } \
NSInvocation *inv = [NSInvocation invocationWithMethodSignature:sig]; \
if (!inv) { [self doesNotRecognizeSelector:sel]; return _return_; } \
[inv setTarget:self]; \
[inv setSelector:sel]; \
va_list args; \
va_start(args, _last_arg_); \
[NSObject setInv:inv withSig:sig andArgs:args]; \
va_end(args);

va_list 等函数

一个C自带的可以用来遍历参数的函数，主要有以下几个： 

va_list,  
va_arg(argp,type),  
va_start,  
va_end

va list主要用于创建一个遍历参数的指针，取参都是通过移动指针来取。原理如下：如下：
void func(int x, float y, char z);
　那么，调用函数的时候，实参 char z 先进栈，然后是 float y，最后是 int x，因此在内存中变量的存放次序是 x->y->z，因此，从理论上说，我们只要探测到任意一个变量的地址，并且知道其他变量的类型，通过指针移位运算，则总可以顺藤摸瓜找到其他的输入变量。
va arg(argp,type) :
　可获取当前valist创建的指针的内容，argp为刚才创建的指针，type为参数类型,每次调用会使指针移动？
va start(argp,par) :
　argp指向传入的第一个可选参数，msg是最后一个确定的参数
va_ end :
结束标记。 

#include <stdio.h>
#include <stdarg.h>

void demo(int num,...)
{
    va_list pt;
    va_start(pt, num);
    if(num) //取第一个参数
    {
        printf("%d\n",num);
    }
    while (1) {
        int number = va_arg(pt, int);
        if (number != -1 ) {
            printf("%d\n",number);
        }
        else{
            va_end(pt);
            break;
        }
    }
}

int main(int argc, const char * argv[]) {
    // insert code here...
    demo(4,5,6,7,8,-1);
    return 0;
}

以上是自己写的一个传多参数的C函数。参考地址

NSMethodSignature和NSInvocation

通常两者相结合使用。官方文档：NSMethodSignature
大致意思表示NSMethodSignature对象能够记录方法的方法类型，参数，返回值信息。想让方法多次转发到多个分散对象中时，可以使用NSMehthodSignature，这种情况下，可以先使用methodSignatureForSelector:方法对方法做签名，然后再创建一个NSInvocation实例与方法签名进行关联。通过NSInvocation的实例对象来对方法的Target，Selector，Argument进行统一设置，最后通过invoke方法进行方法调用。
现在再回头看INIT_INV(sel,nil) 会容易理解很多：
首先传入方法，对方法进行签名和Invoke关联，关联失败进入系统的Error Handle 进行处理。成功则用var_list和Invoke实例进行参数设定。接下来便转入下一个的函数： 

+ (void)setInv:(NSInvocation *)inv withSig:(NSMethodSignature *)sig andArgs:(va_list)args

大致内容就是对参数类型进行分析，因为var_arg函数并不能自己判断类型，所以只能手动进行过滤和判断。对于不支持的参数类型（大型数组等），YY写了一套重试机制，主要是尝试获取更多的长度来确保参数完整性。

+ (BOOL)swizzleInstanceMethod:(SEL)originalSel with:(SEL)newSel

利用ObjRunTime分装的方法交换器对两中方法进行交换。 

+ (BOOL)swizzleInstanceMethod:(SEL)originalSel with:(SEL)newSel {
    Method originalMethod = class_getInstanceMethod(self, originalSel);
    Method newMethod = class_getInstanceMethod(self, newSel);
    if (!originalMethod || !newMethod) return NO;

class_addMethod(self,
                originalSel,
                class_getMethodImplementation(self, originalSel),
                method_getTypeEncoding(originalMethod));
class_addMethod(self,
                newSel,
                class_getMethodImplementation(self, newSel),
                method_getTypeEncoding(newMethod));

method_exchangeImplementations(class_getInstanceMethod(self, originalSel),
                               class_getInstanceMethod(self, newSel));
return YES;

}

class_getInstanceMethod

获取一个Method对象，对象包括一个方法的返回值，SEL,参数等信息。

class_addMethod

这里需要重点说一下的一个方法，它先是添加原Selector,作用是做一层保护，当Class不含这个方法或者对这个方法无法处理的时候，不会进行消息转发，跑到父类去找方法，这是我们不想要的结果。

CFBridge和UUID 

+ (NSString *)deviceUUID {
     NSString *appInfoAccount = @"key_app_info_account";
    NSString *keyDeviceID = @"key_device_uuid";

    NSString *resultString = [SSKeychain passwordForService:keyDeviceID account:appInfoAccount];
    if (resultString == nil) {
        CFUUIDRef puuid = CFUUIDCreate( nil );
        CFStringRef uuidStringRef = CFUUIDCreateString( nil, puuid );
        NSString *uuidString = CFBridgingRelease(uuidStringRef);
        CFRelease(puuid);

        if ([uuidString length] > 0) {
            [SSKeychain setPassword:uuidString forService:keyDeviceID account:appInfoAccount];
            resultString = [SSKeychain passwordForService:keyDeviceID account:appInfoAccount];
        }
    }
  return resultString;
}

对比官方给出的一个获取UUID的例子: 

-(NSString*) uuid {  
    CFUUIDRef puuid = CFUUIDCreate( nil );  
    CFStringRef uuidString = CFUUIDCreateString( nil, puuid );  
    NSString * result = (NSString *)CFStringCreateCopy( NULL, uuidString);  
    CFRelease(puuid);  
    CFRelease(uuidString); return [result autorelease];  
}

今天在iOS中获取设备UUID的时候，看到前辈写了上面一串代码，目的应该是用SSHKeyChain先加密获取到的UUID，再获取加密后的UUID字符串。决定花时间就这段代码里面的知识点做个总结。

UUID

在参考了苹果的官方文档后，
按照自己理解来说，UUID是一种用来标识设备的唯一字符串，在做APP统计或者设备识别的时候可以用到，但是每次调用CFUUIDCreate的时候，创建出来的UUID都不用，因此需要使用keychain或者NSUserdefault来保存获取到得UUID。
在UUID之前，苹果一直使用UDID,在iOS 5.0之后被弃用，现在普遍采用官方给出的UUID进行替代。不同APP，UUID也不同。

CFUUIDCreate

CFUUIDCreate ( CFAllocatorRef alloc );

以上代码可以获取一个uuid， 后面参数根据官方文档一般传NULL或者kCFAllocatorDefault。
文档中还有其他三个方法：
CFUUIDCreateFromString
CFUUIDCreateFromUUIDBytes
CFUUIDCreateWithBytes
前两个是给String对象和UUIDBytes对象生成UUID，后一个是用字节方式手动拼接UUID(个人理解)
一般我们应该使用CFUUIDCreate就足够了 

CFUUIDCreateFromString ( CFAllocatorRef alloc,     CFStringRef uuidStr );

以上代码则是将刚才生成的UUIDRef转换为CFStringRef类型的字符串。
第一个参数同上。
由于生成的字符串是CFStringRef类型的，需要转换成OC风格的字符串才能使用。简单来说以CF开头的对象都是Core Foundation的对象，与Objective-C还是有别。
百度了下CFStringRef收获不多，看了下官方文档，大致是说CFString是一种连接Cocoa和C之间的字符类型，可以高效的对字符串进行操作。在CFString和NSString之间进行转换时，就要讲到Toll-Free Bridged Types
意思是说在CF和NS之间数据是可以进行交换的，CFString和NSString就是一种Toll-Free Bridged Type，但是其他比如NSRunLoop等就没有CFRunLoop与之对应。

CFBridge

以上所讲的NS和CF类型间的互相转换，会涉及到内存管理的问题。比如一个CF转换为NS，内存管理到底有ARC接管还是引用计数器来继续管理，是需要开发者来决定的，有以下三个内存管理交接方法：

__bridge

transfers a pointer between Objective-C and Core Foundation with no transfer of ownership.
直接翻译就是在OC和CF之间转换的时候，不变化指针所有者。
比如OC转成CF之后，还是ARC，CF转成OC后，还是需要Release。比如以下字符串的转换代码: 

//NSString 转 CFStringRef
CFStringRef cfString = (CFStringRef) [[NSString alloc] initWithFormat:@"%@", string];
//...
CFRelease(aCFString);


//CFStringRef 转 NSString
CFStringRef aCFString = CFStringCreateWithCString(kCFAllocatorDefault,
                                                  bytes,
                                                  NSUTF8StringEncoding);
NSString *aNSString = (NSString *)aCFString;
//...
[aNSString release];

__bridge_retained or CFBridgingRetain

casts an Objective-C pointer to a Core Foundation pointer and also transfers ownership to you.
You are responsible for calling CFRelease or a related function to relinquish ownership of the object.
当把一个OC对象转换成CF对象后，放弃ARC管理，移交手动管理内存，需要对对象进行Retain和Release操作
示例代码： 

NSString *aNSString = @"sfsfdsf";    
CFStringRef aCFString = (__bridge_retained CFStringRef) aNSString;  
(CFRelease)aCFString;

__bridge_transfer or CFBridgingRelease

moves a non-Objective-C pointer to Objective-C and also transfers ownership to ARC.
ARC is responsible for relinquishing ownership of the object.
将非OC对象转换为OC对象，同时将对象的管理权交给ARC，开发者无需手动管理内存。如下，不需要手动释放内存。 

NSString *aNSString = @"sfsfdsf";    
CFStringRef aCFString = (__bridge_retained CFStringRef) aNSString;  
//(CFRelease)aCFString;

最后

现在再回头看开头的代码，就会明白不少，首先生成一个UUID，由于是CF类型，需要通过__bridge来移交内存管理权，待成功取得UUID的字符串后，就可以释放CFUUIDRef。最后使用SSKeychain进行安全封装。有机会的话SSKeychain再进行单独介绍。
