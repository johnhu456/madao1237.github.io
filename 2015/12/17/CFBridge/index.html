YOU'VE MADE A BRAVE DECISION, WELCOME.

每一个不曾起舞的日子都是对生命的辜负。

2015-12-17

CFBridge和UUID

+ (NSString *)deviceUUID {
     NSString *appInfoAccount = @"key_app_info_account";
    NSString *keyDeviceID = @"key_device_uuid";

    NSString *resultString = [SSKeychain passwordForService:keyDeviceID account:appInfoAccount];
    if (resultString == nil) {
        CFUUIDRef puuid = CFUUIDCreate( nil );
        CFStringRef uuidStringRef = CFUUIDCreateString( nil, puuid );
        NSString *uuidString = CFBridgingRelease(uuidStringRef);
        CFRelease(puuid);

        if ([uuidString length] > 0) {
            [SSKeychain setPassword:uuidString forService:keyDeviceID account:appInfoAccount];
            resultString = [SSKeychain passwordForService:keyDeviceID account:appInfoAccount];
        }
    }
  return resultString;
}

对比官方给出的一个获取UUID的例子: 

-(NSString*) uuid {  
    CFUUIDRef puuid = CFUUIDCreate( nil );  
    CFStringRef uuidString = CFUUIDCreateString( nil, puuid );  
    NSString * result = (NSString *)CFStringCreateCopy( NULL, uuidString);  
    CFRelease(puuid);  
    CFRelease(uuidString); return [result autorelease];  
}

今天在iOS中获取设备UUID的时候，看到前辈写了上面一串代码，目的应该是用SSHKeyChain先加密获取到的UUID，再获取加密后的UUID字符串。决定花时间就这段代码里面的知识点做个总结。

UUID

在参考了苹果的官方文档后，
按照自己理解来说，UUID是一种用来标识设备的唯一字符串，在做APP统计或者设备识别的时候可以用到，但是每次调用CFUUIDCreate的时候，创建出来的UUID都不用，因此需要使用keychain或者NSUserdefault来保存获取到得UUID。
在UUID之前，苹果一直使用UDID,在iOS 5.0之后被弃用，现在普遍采用官方给出的UUID进行替代。不同APP，UUID也不同。

CFUUIDCreate

CFUUIDCreate ( CFAllocatorRef alloc );

以上代码可以获取一个uuid， 后面参数根据官方文档一般传NULL或者kCFAllocatorDefault。
文档中还有其他三个方法：
CFUUIDCreateFromString
CFUUIDCreateFromUUIDBytes
CFUUIDCreateWithBytes
前两个是给String对象和UUIDBytes对象生成UUID，后一个是用字节方式手动拼接UUID(个人理解)
一般我们应该使用CFUUIDCreate就足够了 

CFUUIDCreateFromString ( CFAllocatorRef alloc,     CFStringRef uuidStr );

以上代码则是将刚才生成的UUIDRef转换为CFStringRef类型的字符串。
第一个参数同上。
由于生成的字符串是CFStringRef类型的，需要转换成OC风格的字符串才能使用。简单来说以CF开头的对象都是Core Foundation的对象，与Objective-C还是有别。
百度了下CFStringRef收获不多，看了下官方文档，大致是说CFString是一种连接Cocoa和C之间的字符类型，可以高效的对字符串进行操作。在CFString和NSString之间进行转换时，就要讲到Toll-Free Bridged Types
意思是说在CF和NS之间数据是可以进行交换的，CFString和NSString就是一种Toll-Free Bridged Type，但是其他比如NSRunLoop等就没有CFRunLoop与之对应。

CFBridge

以上所讲的NS和CF类型间的互相转换，会涉及到内存管理的问题。比如一个CF转换为NS，内存管理到底有ARC接管还是引用计数器来继续管理，是需要开发者来决定的，有以下三个内存管理交接方法：

__bridge

transfers a pointer between Objective-C and Core Foundation with no transfer of ownership.
直接翻译就是在OC和CF之间转换的时候，不变化指针所有者。
比如OC转成CF之后，还是ARC，CF转成OC后，还是需要Release。比如以下字符串的转换代码: 

//NSString 转 CFStringRef
CFStringRef cfString = (CFStringRef) [[NSString alloc] initWithFormat:@"%@", string];
//...
CFRelease(aCFString);


//CFStringRef 转 NSString
CFStringRef aCFString = CFStringCreateWithCString(kCFAllocatorDefault,
                                                  bytes,
                                                  NSUTF8StringEncoding);
NSString *aNSString = (NSString *)aCFString;
//...
[aNSString release];

__bridge_retained or CFBridgingRetain

casts an Objective-C pointer to a Core Foundation pointer and also transfers ownership to you.
You are responsible for calling CFRelease or a related function to relinquish ownership of the object.
当把一个OC对象转换成CF对象后，放弃ARC管理，移交手动管理内存，需要对对象进行Retain和Release操作
示例代码： 

NSString *aNSString = @"sfsfdsf";    
CFStringRef aCFString = (__bridge_retained CFStringRef) aNSString;  
(CFRelease)aCFString;

__bridge_transfer or CFBridgingRelease

moves a non-Objective-C pointer to Objective-C and also transfers ownership to ARC.
ARC is responsible for relinquishing ownership of the object.
将非OC对象转换为OC对象，同时将对象的管理权交给ARC，开发者无需手动管理内存。如下，不需要手动释放内存。 

NSString *aNSString = @"sfsfdsf";    
CFStringRef aCFString = (__bridge_retained CFStringRef) aNSString;  
//(CFRelease)aCFString;

最后

现在再回头看开头的代码，就会明白不少，首先生成一个UUID，由于是CF类型，需要通过__bridge来移交内存管理权，待成功取得UUID的字符串后，就可以释放CFUUIDRef。最后使用SSKeychain进行安全封装。有机会的话SSKeychain再进行单独介绍。
